# Assumptions

## Synthetic Data

This project uses fully synthetic data generated by `src/generate_data.py`. No real port, vessel, carrier, or cargo data is used. The data is designed to be structurally realistic — volume distributions, seasonal patterns, delay correlations, and quality issue rates are modelled after publicly documented port operations benchmarks — but it does not represent any specific real-world facility.

**What the synthetic data gets right:**
- Seasonal weather severity (cosine curve; Northern European winters are 1.5× more severe)
- Vessel class × terminal compatibility constraints (ULCV vessels only call at NCT, the deep-water berth)
- Terminal traffic weights (NCT handles 35% of calls; WIT handles 10%)
- Carrier service rotations (each carrier has a fixed set of vessel IDs they operate)
- Realistic crane-to-call ratios (2–4 cranes per call, 5–50 moves per crane assignment)
- Injected dirty data at approximately 4% rate: 30 missing ETAs, 20 missing ATAs, 10 invalid timestamp sequences, 20 duplicate call records, 8 late-update records

**What the synthetic data simplifies:**
- Delay is modelled as a log-normal random variable with a weather and congestion multiplier, not from actual tidal conditions, berth availability, or pilot schedules
- Carrier and terminal IDs are fictional codes, though they mirror the names of major real carriers
- Container moves are independent draws; real moves have strong autocorrelation within a shift
- Weather is independent across terminals; in reality, terminals in the same region share weather

## Business Logic

**Delay threshold:** the binary ML label (`is_arrival_delayed_more_than_2_hours`) uses a two-hour cutoff. This is an operational convention: delays under 30 minutes are typically absorbed by buffer time; delays of 1–2 hours trigger operational adjustments; delays over 2 hours propagate to downstream legs. The exact threshold is configurable via `DELAY_THRESHOLD_HOURS` in `src/config.py`.

**Vessel class hierarchy:** Sub-Panamax < Panamax < Post-Panamax < ULCV. Larger vessels have higher turnaround time variability and are more sensitive to weather.

**Turnaround hours:** `actual_turnaround_hours = (ATD − ATA)` in hours. The silver layer preserves extreme values (e.g., multi-day calls) with an `excessive_turnaround_flag` rather than capping them, on the grounds that a 7-day call at a repair berth is real, not an error.

**Weather impact:** the `weather_impact_factor` in `weather_daily_silver` is a multiplicative scalar (1.0 = no impact; higher = slower operations). It combines wind, wave, and visibility conditions. The specific weights are domain-informed estimates, not calibrated to real port data.

## ML Modelling

**Point-in-time correctness:** all features must represent information available to an operations analyst at the moment the ETA notification arrives — before the vessel physically reaches the terminal. Any column derived from ATA, ATD, or actual delay is excluded from the feature set and treated as target leakage.

**Imputation strategy:** rolling history features (`avg_previous_10_terminal_delays`, `previous_vessel_delay`) are null for calls early in the simulation window that have no qualifying prior history. These are imputed at training time with the column median via `SimpleImputer`. At inference time, calls without history receive the same median — a pragmatic choice that slightly underestimates uncertainty for new vessels and terminals.

**Class imbalance:** 26.6% of labeled calls exceed the 2-hour delay threshold. Both models use `class_weight="balanced"`, which up-weights the minority class inversely proportional to its frequency. This improves recall for delayed calls at some cost to precision — the appropriate trade-off for an operations alert system where missing a delay is more costly than a false alarm.

**Model metrics interpretation:** the AUC values (LR: 0.477, RF: 0.538) are near-random on this dataset. This is expected — the synthetic delay variable is generated with substantial randomness that is intentionally not explained by the available features. The project is designed to demonstrate the feature engineering, PIT correctness, and pipeline architecture, not to achieve high predictive accuracy on artificial data. On real port operations data, where delays have genuine structural causes (tidal windows, pilot availability, berth congestion), these same features would produce meaningfully higher AUC values.

**No hyperparameter tuning:** the models use fixed hyperparameters (`n_estimators=300`, `min_samples_leaf=10`, `C=0.5`). A production deployment would include cross-validated grid search or Bayesian optimisation. This is omitted to keep the focus on the data pipeline and feature engineering decisions rather than model selection mechanics.

## Infrastructure

**Single-machine execution:** the full pipeline runs on a single laptop in under 60 seconds. It is not distributed. The architecture is designed to be directly translatable to a Spark/Delta Lake or Databricks environment — the medallion layer contracts, the PIT join pattern, and the DQ flag approach all port directly — but no distributed execution framework is included in this repository.

**No real-time component:** this is a batch pipeline. It simulates the nightly or hourly batch refresh pattern common in port analytics platforms, not a streaming architecture. The ETA prediction use case in a production system would be triggered by incoming AIS (Automatic Identification System) messages or port system API calls.
